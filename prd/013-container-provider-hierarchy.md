# PRD 013: ContainerProvider Hierarchy and ShellExecutor

## Problem Statement

Knox's `AgentProvider` interface was extracted in PRD 012 to decouple the
engine from Claude-specific concerns. However, the abstraction is
unvalidated — only one implementation exists. Simultaneously, Knox needs
a way to run one-shot container jobs (PR summary generation, utility
scripts) without the LLM loop machinery (sentinel detection, check
commands, re-prompting). PRD 011 proposed a separate `UtilityContainer`
abstraction, but that design predates the `AgentProvider` extraction and
would duplicate container lifecycle management.

The current flat `AgentProvider` interface forces all providers to accept
LLM-specific context (`loopNumber`, `maxLoops`, `checkFailure`) even when
it's irrelevant — a shell command doesn't loop. The type system can't
distinguish which provider needs what context, and the caller must
populate dead fields.

## Solution

Introduce a two-tier provider hierarchy rooted at
`ContainerProvider<TContext>`. LLM agents and shell commands are sibling
branches beneath this root, each with their own context type and runner.
This validates the provider abstraction with a concrete second
implementation, eliminates the need for a separate `UtilityContainer`
abstraction, and lets the type system enforce which runner can use which
provider.

One-shot LLM calls (e.g., generating a PR summary by invoking the Claude
CLI with a prompt) are shell-tier jobs — invoking a model binary is a
shell command, not an agent loop. The distinction between shell and agent
is about orchestration strategy (single invoke vs. loop with sentinel
detection), not about whether an LLM is involved.

## User Stories

1. As a Knox maintainer, I want a type hierarchy that separates
   container-level concerns from LLM-specific concerns, so that the type
   system prevents passing irrelevant context to providers.

2. As a Knox maintainer, I want a `ShellProvider` that runs a single
   command in a container and returns a completion status, so that
   utility jobs reuse the existing container lifecycle without
   duplicating it.

3. As a Knox maintainer, I want a `ShellExecutor` that runs a shell
   provider once without looping, retrying, or commit-nudging, so that
   one-shot jobs have a simple orchestration path.

4. As a Knox maintainer, I want `ClaudeCodeAgentProvider` to implement
   the LLM-specific tier of the hierarchy, so that the refactored types
   are validated against the existing working implementation.

5. As a Knox maintainer, I want the `AgentRunner` to require the
   LLM-specific provider tier, so that only loop-capable providers can
   be passed to the loop runner.

6. As a Knox maintainer, I want the existing agent runner and provider
   tests to pass against the refactored types, so that the rename
   doesn't break existing behavior.

7. As a Knox maintainer, I want unit tests for `ShellProvider` and
   `ShellExecutor` using mock container handles, so that the new
   modules are verified in isolation.

8. As a Knox maintainer, I want an integration smoke test that runs a
   shell command in a real container via `ShellExecutor`, so that the
   wiring is validated end-to-end.

9. As a Knox user running queues, I want PR summaries generated by a
   one-shot Claude CLI call via the shell tier, so that PRs get
   meaningful titles and descriptions without a separate utility
   container abstraction.

10. As a Knox maintainer, I want future LLM providers (Codex, Aider) to
    slot into the `AgentProvider` branch without touching the shell tier,
    so that the hierarchy supports multiple agents cleanly.

11. As a Knox maintainer, I want the `ContainerContext` base to carry the
    `onLine` callback, so that both shell commands and LLM agents can
    stream output to the TUI.

12. As a Knox maintainer, I want `InvokeResult` to remain
    `{ completed: boolean, exitCode: number }` for both tiers initially,
    so that the return type stays simple until a consumer needs richer
    data.

## Implementation Decisions

### Type hierarchy

```
ContainerContext                     — container: ContainerHandle, onLine?
  ├─ ShellContext                    — command: string
  └─ LlmAgentContext                — task, loopNumber, maxLoops, checkFailure?, customPrompt?

ContainerProvider<TContext extends ContainerContext>
  invoke(ctx: TContext): Promise<InvokeResult>

ShellProvider             implements ContainerProvider<ShellContext>
AgentProvider<T extends LlmAgentContext> extends ContainerProvider<T>
  └─ ClaudeCodeAgentProvider implements AgentProvider<LlmAgentContext>
```

- `ContainerProvider` is the generic root — "thing that does work in a
  container."
- `AgentProvider` is the LLM-specific tier — "thing that acts as an
  autonomous agent." It is a marker interface that narrows the generic
  bound; it adds no new methods.
- `ShellProvider` sits alongside `AgentProvider`, not under it.

### Context types

- `ContainerContext` holds `container: ContainerHandle` and
  `onLine?: (line: string) => void`. These are container-level concerns
  shared by all providers.
- `ShellContext` extends `ContainerContext` with `command: string`. The
  shell provider runs `sh -c ${command}`.
- `LlmAgentContext` extends `ContainerContext` with `task: string`,
  `loopNumber: number`, `maxLoops: number`, `checkFailure?: string`,
  `customPrompt?: string`. These are loop-aware fields the agent runner
  populates on each iteration.

### Runners

- `AgentRunner` (existing) handles the LLM loop: invoke, check, re-invoke
  with failure context, commit nudge. Requires
  `AgentProvider<LlmAgentContext>`.
- `ShellExecutor` (new) handles single-invoke shell jobs: invoke once,
  return result. Requires `ContainerProvider<ShellContext>`. No loop, no
  retry, no commit nudge.

### Return type

- `InvokeResult` stays as `{ completed: boolean, exitCode: number }` for
  both tiers. Evolve to a discriminated union when a consumer needs
  richer data (e.g., captured stdout, structured output).

### Provider wiring

- Direct construction. The orchestrator builds
  `new ClaudeCodeAgentProvider(model)` for queue items and
  `new ShellProvider()` for utility jobs. No factory or registry — the
  orchestrator already knows which code path it's on.
- A factory/registry is deferred until multiple LLM providers exist and
  the orchestrator needs config-driven provider selection.

### Naming

- `AgentProvider` name is preserved for the LLM tier — it's the
  established term from ADR 001 and the codebase.
- `AgentRunner` name is preserved — it's entrenched and its meaning is
  clear in context.
- The current `AgentContext` is renamed to `LlmAgentContext`.
- The current `AgentProvider` interface is renamed to
  `ContainerProvider` (base) with `AgentProvider` becoming the LLM tier.

### PRD 011 superseded

- The `UtilityContainer` concept from PRD 011 is replaced by the shell
  tier. Shell jobs are the engine with different wiring, not a separate
  abstraction.
- PR summary generation (the first consumer from PRD 011) becomes a
  `ShellProvider` job: run the Claude CLI one-shot in a container,
  capture stdout for PR title and body.

### Modules to modify

- **Container provider types** — split `agent_provider.ts` into the new
  type hierarchy.
- **ClaudeCodeAgentProvider** — update to implement
  `AgentProvider<LlmAgentContext>`.
- **AgentRunner** — update to require `AgentProvider<LlmAgentContext>`.
- **Knox engine** — update provider construction and type references.
- **mod.ts exports** — export new types.
- **Existing tests** — update for new type names.

### New modules

- **ShellProvider** — implements `ContainerProvider<ShellContext>`.
- **ShellExecutor** — single-invoke runner.
- **ShellProvider + ShellExecutor tests** — unit tests with mock
  `ContainerHandle`.

### Implementation sequence

1. Refactor types — split contexts, rename interfaces, update all
   consumers. Existing tests must pass.
2. Build `ShellProvider` + `ShellExecutor` with unit tests.
3. Integration smoke test — `ShellExecutor` + `ShellProvider` running
   `echo hello` in a real container.
4. PR summary generation — first real consumer via the shell tier.
5. Difficulty-based model selection (issue #8) — layer on top once the
   provider hierarchy is stable.

## Testing Decisions

Good tests verify external behavior through the public interface, not
implementation details. Mock at the `ContainerHandle` boundary — that's
the seam between Knox and the container runtime.

### Modules to test

- **ShellProvider** — unit tests with mock `ContainerHandle`. Verify:
  command is passed to `sh -c`, exit 0 maps to `completed: true`,
  non-zero maps to `completed: false`, `onLine` callback receives
  output.
- **ShellExecutor** — unit tests with mock `ContainerProvider<ShellContext>`.
  Verify: single invoke (not looped), result forwarded, no commit nudge.
- **AgentRunner** (existing tests updated) — verify new type names
  compile and existing behavior is preserved.
- **ClaudeCodeAgentProvider** (existing tests updated) — verify new type
  names compile and existing behavior is preserved.
- **Integration smoke test** — `ShellExecutor` + `ShellProvider` +
  `DockerRuntime` running `echo hello` in a real container, asserting
  `completed: true`.

### Prior art

- `test/agent/agent_runner_test.ts` — mock provider pattern, covers loop
  behavior, check commands, commit nudge, abort.
- `test/agent/claude_code_agent_provider_test.ts` — mock
  `ContainerHandle`, covers prompt building, sentinel detection,
  streaming.

## Out of Scope

- Difficulty-based model selection (issue #8) — orthogonal, sequenced
  after this work.
- Provider factory/registry — deferred until multiple LLM providers
  exist.
- Discriminated union return type — deferred until a consumer needs
  richer data.
- Second LLM provider (Codex, Aider) — this work validates the
  abstraction; building a second LLM provider is separate.
- PR summary generation implementation — sequenced as step 4 after this
  PRD's core work is done. The shell tier enables it; this PRD builds
  the tier.
